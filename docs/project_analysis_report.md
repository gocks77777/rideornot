# 🚀 RideOrNot (탈래말래) 프로젝트 종합 분석 및 개선 리포트

본 문서는 현재 `rideornot` 프로젝트의 상태를 **사용자 경험(UX), 보안, 서버 및 아키텍처** 3가지 핵심 관점에서 깊이 있게 분석하고, 앞으로 해결해야 할 과제와 우선순위를 정리한 보고서입니다.

---

## 🚨 핵심 문제점 및 분석

### 1. 보안 (Security) & 환경 변수 관리
- **현재 상황:** Vercel 배포 과정에서 확인했듯, Supabase (DB) 및 카카오/네이버 서드파티 API 키들을 다수 사용 중입니다.
- **문제점:**
  - `NEXT_PUBLIC_` 환경변수가 클라이언트에 노출되는 것은 정상이나, Supabase의 `anon_key`가 브라우저에 노출될 때 **RLS(Row Level Security)** 설정이 엄격하지 않으면 악의적인 사용자가 DB 데이터를 조작할 수 있습니다.
  - 현재 `parties`, `party_members`, `users` 등의 테이블에 대한 권한 제어(RLS)가 명확히 설정되어 있는지 점검이 필요합니다.

### 2. 서버 및 아키텍처 (Server & Architecture)
- **현재 상황:** Next.js (App Router) 기반으로 Vercel에 배포되어 있으며, Supabase를 BaaS(Backend as a Service)로 활용 중입니다.
- **문제점:**
  - **API 라우트 최적화:** `app/api/geocode/route.ts` 등에서 네이버 API를 호출할 때 에러 핸들링은 있으나, Timeout이나 재시도(Retry) 로직이 부족하여 외부 API 장애 시 서비스 전체 경험이 저하될 수 있습니다.
  - **실시간 데이터 동기화 (Realtime):** Supabase의 Realtime 기능(`comments` 등)을 사용 중인데, 연결이 끊기거나 폴링이 지연될 때의 예외 처리가 필요합니다. (`RealtimeClient.js` 관련 Webpack 경고 로그가 지속적으로 발생하고 있음)
  - **캐싱 및 서버 과부하:** 사용자가 늘어날수록 메인 화면의 팟 목록(`parties`) 조회 쿼리가 무거워질 수 있습니다. Next.js의 캐싱(Cache) 메커니즘이나 ISR(Incremental Static Regeneration) 도입이 고려되지 않은 상태입니다.

### 3. 사용자 경험 (UX / UI)
- **현재 상황:** 카카오 로그인, 네이버 지도 표시, 검색어 자동완성 등 주요 기능이 연동되어 있습니다.
- **문제점:**
  - **오류 피드백 부재:** 네이버 지도 스크립트 로드 실패, 또는 검색 API 오류 시 빈 화면이 나오거나 무한 로딩이 걸릴 위험이 있습니다. 사용자에게 "현재 지도 서버에 문제가 있습니다" 등의 친절한 안내가 필요합니다.
  - **초기 로딩 속도:** 외부 스크립트(카카오, 네이버)와 폰트, 무거운 라이브러리가 동시에 로드되며 초기 렌더링(LCP)이 지연될 수 있습니다.
  - **상태 동기화 지연:** 다른 사용자가 팟에 참여했을 때, 새로고침을 하지 않으면 내 화면에 즉시 반영되지 않는 문제(Optimistic UI 부재)가 있을 수 있습니다.

## 🎯 해결 방안 및 액션 플랜 (우선순위 순)

### 🔴 1순위: 치명적 보안 및 데이터 무결성 확보 (즉각 조치)
1. **Supabase RLS (Row Level Security) 설정 강화**
   - **조치:** `users`, `parties`, `party_members` 테이블에 대해 "자신의 데이터만 수정 가능", "로그인한 사용자만 읽기 가능" 등의 정책(Policy)을 SQL로 엄격하게 정의해야 합니다.
   - **이유:** 익명 키(`anon_key`)가 공개되어 있으므로, RLS가 없으면 누구나 남의 팟을 지우거나 계좌번호를 탈취할 수 있습니다.

2. **환경 변수 분리 및 검증**
   - **조치:** `zod`나 `joi` 같은 라이브러리를 사용하여, 앱 실행 시 필수 환경 변수(네이버, 카카오 등)가 없으면 즉시 에러를 뱉고 빌드를 멈추도록(Fail-fast) 개선해야 배포 실수를 막을 수 있습니다.

### 🟡 2순위: 핵심 사용자 경험(UX) 개선 및 에러 핸들링
3. **API 예외 처리 및 로딩/에러 UI 추가**
   - **조치:** `app/api/geocode/route.ts` 호출 시 실패하면 `Toast` 알림이나 에러 메시지를 명확히 표시합니다.
   - **조치:** React Suspense와 Error Boundary를 적극 도입하여, 지도나 리스트가 로딩 중일 때 '스켈레톤(Skeleton)' UI를 보여주어 사용자 이탈을 막습니다.

4. **Optimistic UI (낙관적 업데이트) 적용**
   - **조치:** '팟 참여하기'나 '댓글 달기' 버튼을 눌렀을 때, 서버의 응답을 기다리지 않고 즉시 화면에 결과를 반영한 뒤 백그라운드에서 동기화하도록 UI를 개선합니다.

### 🟢 3순위: 성능 최적화 및 아키텍처 고도화 (장기 과제)
5. **Next.js Caching 및 쿼리 최적화**
   - **조치:** 자주 변하지 않는 데이터는 캐싱하고, 실시간 팟 목록은 `SWR` 이나 `React Query`를 도입해 캐싱+실시간 폴링을 결합하여 Supabase DB 부하를 줄입니다.

6. **외부 스크립트 최적화 (Third-party Scripts)**
   - **조치:** `next/script`의 `strategy="lazyOnload"` 또는 `worker`를 활용하여 카카오 로그인 SDK나 네이버 지도 API가 앱의 초기 렌더링 속도를 깎아먹지 않도록 지연 로딩 처리합니다.

---

## 📝 결론 및 제언
현재 **RideOrNot** 프로젝트는 MVP(최소 기능 제품)로써 핵심 로직(로그인, 지도, 팟 매칭)은 아주 훌륭하게 구현되어 있습니다. 
하지만 실제 수백, 수천 명의 유저가 사용하기 시작하면 **DB 보안(RLS)**과 **네트워크 지연 시의 UX 처리**에서 문제가 터질 확률이 높습니다. 

따라서 가장 먼저 **Supabase 대시보드에 들어가서 각 테이블의 RLS 정책이 안전하게 걸려있는지 확인(1순위)**하는 것을 추천드립니다. 이후 프론트엔드의 로딩/에러 UI를 다듬어가는 순서로 개발을 진행하시면 훨씬 탄탄하고 성공적인 서비스를 만들 수 있습니다!
