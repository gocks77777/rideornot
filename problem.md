# 탈래말래? — 전국 단위 서비스 확장 시 문제점 분석

> 백엔드/프론트엔드/보안 전문가 관점의 종합 진단서
> 대한민국 전 국민(5,100만 명) 대상 홍보 시 발생 가능한 모든 문제점

---

## 🔴 CRITICAL (서비스 장애 직결)

### C-1. Rate Limiting 완전 부재
- **현재 상태**: 모든 API Route(`/api/praise`, `/api/push`, `/api/directions`, `/api/geocode`, `/api/admin/action`, `/api/delete-account`)에 Rate Limiting이 없음
- **위험**: 악의적 사용자가 초당 수천 건 요청 가능 → Supabase 무료/Pro 플랜 연결 풀 고갈, Naver API 일일 할당량 초과, Vercel 함수 동시 실행 한도 초과
- **시나리오**: 봇이 `/api/geocode`를 초당 100회 호출 → Naver API 차단 → 전체 사용자 검색 불가
- **해결**: `upstash/ratelimit` 또는 Vercel Edge Middleware에서 IP/유저별 Rate Limit 적용

### C-2. Supabase 연결 풀 한계
- **현재 상태**: Supabase Pro 플랜 기준 직접 연결 60개, Pooler 200개
- **위험**: 동시 접속자 1,000명만 되어도 연결 풀 초과 → 502/503 에러 폭주
- **시나리오**: 출퇴근 피크(08:00~09:00) 동시 접속 5,000명 → DB 연결 불가 → 앱 전체 마비
- **해결**: Supabase Pooler(Transaction mode) 필수 적용, 연결 풀 모니터링 설정, 필요 시 Enterprise 플랜 또는 자체 PostgreSQL + PgBouncer

### C-3. Realtime 구독 폭발
- **현재 상태**: `app/page.tsx`에서 모든 클라이언트가 `parties` + `party_members` + `comments` 테이블에 Realtime 구독
- **위험**: Supabase Realtime은 동시 연결 수 제한 있음 (Pro: 500 concurrent). 전국 사용자 접속 시 즉시 한도 초과
- **시나리오**: 10,000명 동시 접속 → Realtime 연결 거부 → 실시간 업데이트 중단 → 사용자 혼란
- **해결**: Realtime 구독을 선택적으로 (현재 보고 있는 팟만 구독), 목록은 polling으로 전환, 또는 자체 WebSocket 서버 구축

### C-4. 단일 `app/page.tsx` (822줄) 성능 붕괴
- **현재 상태**: 모든 상태 관리, 데이터 페칭, 비즈니스 로직이 하나의 클라이언트 컴포넌트에 집중
- **위험**: 모든 상태 변경마다 전체 리렌더링 → 저사양 모바일 기기에서 프레임 드랍, 메모리 누수
- **시나리오**: 팟 100개 + 댓글 1,000개 로드 시 → 렌더링 3초+ → 사용자 이탈
- **해결**: 상태 관리 분리(Zustand/Jotai), React.memo 적용, 가상 스크롤(react-window), 서버 컴포넌트 활용

### C-5. 페이지네이션 없음
- **현재 상태**: `fetchPods()`가 모든 팟을 한 번에 SELECT → 클라이언트에서 필터링
- **위험**: 전국 서비스 시 일일 팟 생성 수만 건 → 한 번에 수만 건 로드 시도 → 응답 시간 10초+ → OOM
- **시나리오**: 서울에서만 하루 5,000개 팟 생성 → 전체 목록 요청 시 타임아웃
- **해결**: 서버 사이드 페이지네이션 + 커서 기반 무한 스크롤, 위치 기반 필터링을 DB 쿼리 레벨에서 수행

## 🟠 HIGH (사용자 피해 및 법적 리스크)

### H-1. 개인정보보호법(PIPA) 위반 리스크
- **현재 상태**: 카카오 닉네임, 프로필 사진, 성별, 은행 계좌번호를 수집하나, 개인정보 처리방침이 형식적
- **위험**: 전국 서비스 = 정보통신서비스 제공자 → 개인정보보호위원회 규제 대상
- **필수 조치**:
  - 개인정보 수집·이용 동의 절차 (opt-in)
  - 개인정보 처리방침 법정 필수 항목 모두 기재
  - 만 14세 미만 이용자 법정대리인 동의
  - 개인정보 보호책임자(CPO) 지정 및 공개
  - 개인정보 파기 절차 명시 (탈퇴 후 보관 기간)
  - 제3자 제공 동의 (카카오, Supabase 해외 서버)

### H-2. 위치정보법 위반
- **현재 상태**: GPS 위치 수집 + 출발/도착 좌표 저장 → 위치정보 사업자 신고 필요
- **위험**: 위치정보의 보호 및 이용 등에 관한 법률 위반 시 3년 이하 징역 또는 3천만원 이하 벌금
- **필수 조치**:
  - 위치기반서비스 사업자 신고 (방송통신위원회)
  - 위치정보 수집·이용 동의 별도 획득
  - 위치정보 이용·제공사실 확인자료 보관 (6개월)

### H-3. 여객자동차운수사업법 리스크
- **현재 상태**: 택시 합승 매칭 서비스 → 법적 회색지대
- **위험**: 2024년 택시 합승 합법화 법안 통과 여부와 무관하게, 유상 운송 중개로 해석될 여지
- **필수 조치**:
  - 법률 자문을 통한 서비스 구조 검토
  - 약관에 "택시 탑승 비용 정산 보조 도구"로 명시
  - 불법 유상 운송 중개가 아님을 명확히 하는 면책 조항

### H-4. 전자금융거래법 리스크
- **현재 상태**: 보증금(deposit) 수취 및 정산 기능 → 전자지급결제대행(PG) 해당 가능
- **위험**: 무허가 전자금융업 → 5년 이하 징역 또는 5천만원 이하 벌금
- **필수 조치**:
  - 직접 정산 중개하지 않도록 구조 변경 (외부 송금 링크만 제공)
  - 또는 PG사 연동 (토스페이먼츠, 카카오페이)
  - 에스크로(예치) 서비스 도입 검토

### H-5. 결제/정산 사기 취약
- **현재 상태**: 호스트가 수동으로 입금 확인 → 조작 가능, 환불 메커니즘 없음, 거래 로그 없음
- **위험**: "입금했다" 거짓 주장, 호스트가 확인 후 팟 취소, 보증금 먹튀
- **시나리오**: 악의적 호스트가 보증금 10,000원 설정 → 4명 입금 → 팟 취소 → 40,000원 편취
- **해결**: 에스크로 시스템, 자동 환불 로직, 거래 내역 테이블, 분쟁 해결 프로세스

### H-6. 성범죄/안전 사고 책임
- **현재 상태**: 성별 필터, 매너 점수, 신고 시스템만 존재 → 실질적 신원 확인 없음
- **위험**: 낯선 사람과의 합승 → 성범죄, 폭행, 납치 사건 발생 시 플랫폼 책임 논란
- **필수 조치**:
  - 본인 인증 (PASS, 카카오 본인인증)
  - 긴급 신고 버튼 (112 연결)
  - 탑승 중 실시간 위치 공유 옵션
  - 사고 발생 시 대응 프로세스 및 보험 검토
  - 면책 조항 + 이용약관 강화

### H-7. XSS/인젝션 공격 벡터
- **현재 상태**: 댓글(`message`), 출발/도착 텍스트(`start_point`, `end_point`), 신고 사유(`detail`)가 HTML 이스케이프 없이 렌더링
- **위험**: `<script>` 또는 `<img onerror>` 삽입 → 다른 사용자 세션 탈취
- **시나리오**: 악성 댓글로 다른 사용자의 Supabase 토큰 탈취 → 계정 도용
- **해결**: DOMPurify로 출력 시 sanitize, CSP(Content-Security-Policy) 헤더 설정

### H-8. Supabase Service Role Key 노출 위험
- **현재 상태**: `SUPABASE_SERVICE_ROLE_KEY`가 서버 사이드에서만 사용되지만, `.env.local`이 레포에 포함된 적 있음
- **위험**: Service Role Key = RLS 우회 → 모든 테이블 읽기/쓰기/삭제 가능
- **해결**: 즉시 키 로테이션, `.env.local`을 `.gitignore`에 확인, Git 히스토리에서 제거 (`git filter-branch` 또는 BFG)

## 🟡 MEDIUM (사용자 경험 저하 및 운영 비용)

### M-1. 에러 모니터링 부재
- **현재 상태**: `console.error`만 사용, Sentry/LogRocket 미도입
- **위험**: 전국 서비스에서 발생하는 에러를 실시간 파악 불가 → 장애 인지까지 수 시간 소요
- **해결**: Sentry 도입 (무료 5,000 에러/월), 슬랙 알림 연동

### M-2. CDN/캐싱 전략 부재
- **현재 상태**: 모든 API 요청이 직접 Supabase/Naver API 호출
- **위험**: 동일 경로 검색이 반복 → 불필요한 API 비용, 응답 지연
- **해결**: Naver 길찾기 결과 캐싱 (Redis/Vercel KV), 정적 자산 CDN, API 응답 Cache-Control 헤더

### M-3. Naver API 의존도 + 장애 전파
- **현재 상태**: 검색, 지오코딩, 길찾기 모두 Naver API 의존 → 3초 타임아웃
- **위험**: Naver API 장애 시 팟 생성/검색 전체 불가
- **해결**: 카카오맵 API를 fallback으로 추가, 자주 검색되는 경로 캐싱, 타임아웃 시 graceful degradation

### M-4. 택시 요금 추정 부정확
- **현재 상태**: `4800 + (km - 1.6) * 1300` 고정 공식 (2025 기준)
- **위험**: 야간 할증(20%), 시외 요금, 지역별 요금 차이 미반영 → 사용자 불신
- **해결**: Naver Directions API의 `taxiFare` 값을 우선 사용 (이미 호출 중), 시간대별 할증 반영

### M-5. 푸시 알림 전송 실패 시 재시도 없음
- **현재 상태**: `web-push`로 1회 전송 시도, 실패 시 로그만
- **위험**: 중요 알림(팟 출발, 참여 승인) 누락 → 사용자 노쇼 증가
- **해결**: 알림 큐 시스템(Bull/BullMQ), 실패 시 3회 재시도, 알림 히스토리 테이블

### M-6. 검색/필터 기능 부족
- **현재 상태**: 출발지/도착지 텍스트 매칭만 존재, 반경 검색 없음
- **위험**: 전국 서비스 시 수만 개 팟 → 원하는 팟 찾기 어려움 → 사용자 이탈
- **해결**: PostGIS 기반 반경 검색, 시간대 필터, 경유지 검색, 실시간 정렬(거리/출발시간)

### M-7. 이미지/파일 업로드 검증 없음
- **현재 상태**: 카카오 프로필 이미지 URL을 그대로 사용, 자체 이미지 업로드 기능 없음
- **위험**: 향후 이미지 업로드 추가 시 악성 파일 업로드 가능
- **현재 리스크**: `avatar_url`에 악성 URL 저장 가능 (`safeAvatarUrl` 추가했으나, CSP 없이는 불충분)
- **해결**: CSP 헤더로 이미지 출처 제한, Supabase Storage 사용 시 파일 타입/크기 검증

### M-8. 서비스 워커(SW) 업데이트 전략 없음
- **현재 상태**: `sw.js`가 push 이벤트만 처리, 버전 관리/캐시 무효화 로직 없음
- **위험**: 앱 업데이트 후에도 사용자가 구버전 캐시 사용 → 기능 불일치, 버그 지속
- **해결**: SW 버전 관리, `skipWaiting` + `clients.claim`, 앱 업데이트 알림 UI

### M-9. Next.js 13.5.1 보안 취약점
- **현재 상태**: 2023년 9월 버전 → 알려진 보안 패치 미적용
- **위험**: CVE 공개 취약점 악용 가능
- **해결**: Next.js 14.x 이상으로 업그레이드 (App Router 호환성 테스트 필요)

### M-10. 관리자 대시보드 확장성 부족
- **현재 상태**: 신고 목록만 조회, 페이지네이션 없음 (limit 200)
- **위험**: 전국 서비스 시 일일 신고 수백 건 → 관리 불가
- **해결**: 필터/검색/페이지네이션, 자동 처리 규칙 (동일 사용자 3회 신고 → 자동 정지), 대시보드 통계

## 🟢 LOW (장기적 개선 사항)

### L-1. 테스트 코드 전무
- **현재 상태**: 단위/통합/E2E 테스트 없음
- **위험**: 코드 변경 시 사이드 이펙트 감지 불가 → 프로덕션 장애 빈도 증가
- **해결**: Vitest (단위), Playwright (E2E), CI/CD 파이프라인에서 자동 실행

### L-2. CI/CD 파이프라인 없음
- **현재 상태**: 수동 배포 (git push → Vercel/Netlify 자동 빌드만)
- **위험**: 빌드 실패, 타입 에러, 린트 에러가 프로덕션 반영 가능
- **해결**: GitHub Actions → lint + type-check + test + build → preview deploy → 수동 승인 → production

### L-3. 로깅/감사 추적 없음
- **현재 상태**: API 호출 로그, 관리자 행위 로그, 사용자 활동 로그 없음
- **위험**: 사고 발생 시 원인 파악 불가, 법적 분쟁 시 증거 부족
- **해결**: 구조화된 로깅 (Pino/Winston), 관리자 행위 감사 로그 테이블, Supabase Database Webhooks

### L-4. 접근성(a11y) 미준수
- **현재 상태**: ARIA 라벨 부족, 키보드 네비게이션 미지원, 색상 대비 미검증
- **위험**: 장애인차별금지법 위반 가능, 웹 접근성 인증 마크 획득 불가
- **해결**: lighthouse 접근성 감사, ARIA 속성 추가, 키보드 포커스 관리

### L-5. 국제화(i18n) 미지원
- **현재 상태**: 모든 텍스트가 한국어 하드코딩
- **위험**: 외국인 거주자/관광객 이용 불가
- **해결**: next-intl 또는 react-i18next 도입

### L-6. 데이터 백업/복구 전략 없음
- **현재 상태**: Supabase 자동 백업에만 의존
- **위험**: 실수로 데이터 삭제 시 복구 시간 불확실
- **해결**: 일일 자동 백업 + 복구 테스트, Point-in-Time Recovery 활성화

### L-7. 멀티 디바이스 세션 관리 없음
- **현재 상태**: 같은 계정으로 여러 디바이스 로그인 시 충돌 가능
- **위험**: 동시에 팟 참여/취소 등 동작 시 데이터 불일치
- **해결**: 디바이스별 세션 관리, 활성 세션 목록 제공

### L-8. 앱 스토어 배포 미비
- **현재 상태**: PWA만 지원 → iOS Safari에서 푸시 알림 제한적
- **위험**: iOS 사용자(한국 점유율 ~30%) 기능 제한 → 사용자 이탈
- **해결**: Capacitor/TWA로 네이티브 앱 빌드 → App Store/Play Store 등록

## 최종 진단

> **현재 상태**: 학교/소규모 커뮤니티 내 MVP로는 충분하지만, **전국 단위 서비스로 홍보하기에는 법적·인프라·보안 측면에서 준비가 부족합니다.**

**가장 큰 3가지 리스크:**

1. **법적 리스크** — 개인정보보호법, 위치정보법, 전자금융거래법 위반 가능성. 서비스 중지 명령 또는 형사 처벌 대상이 될 수 있음
2. **인프라 리스크** — Rate Limiting 없음 + 페이지네이션 없음 + Realtime 한도 → 동시 접속 1,000명 이상에서 서비스 장애 발생 확실
3. **안전 리스크** — 본인 인증 없는 합승 매칭 → 사건 발생 시 사회적 비난 + 플랫폼 책임 논란

**권장 단계:**
1. **1단계 (1~2주)**: Rate Limiting + 페이지네이션 + 에러 모니터링 + CSP 헤더
2. **2단계 (2~4주)**: 법률 자문 + 개인정보 처리방침 정비 + 위치정보 사업자 신고 + 본인인증
3. **3단계 (1~2개월)**: 결제 시스템 정비 + 반경 검색 + 테스트 코드 + CI/CD
4. **4단계 (2~3개월)**: 네이티브 앱 배포 + 관리자 도구 강화 + 대규모 부하 테스트
